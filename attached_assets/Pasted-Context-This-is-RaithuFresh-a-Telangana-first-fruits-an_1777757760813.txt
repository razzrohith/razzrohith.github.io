Task:
Apply a small security-hardening patch based on the latest QA report.

Current QA findings:
1. public_insert_listings on produce_listings allows anon listing insert.
2. public_update_call_request_status on agent_call_requests allows anon status update.
3. auth_read_call_requests currently allows authenticated users to SELECT all agent call requests.

Goal:
Harden these policies now that authentication and role-based routes exist.

Step 1 — Create SQL patch

Create:

supabase/patch-post-qa-security-hardening.sql

Required changes:

A. produce_listings

Remove anonymous insert access:
- Drop public_insert_listings policy if it exists.
- Revoke INSERT on produce_listings from anon if currently granted.

Keep:
- anon can still SELECT active produce_listings.
- authenticated farmers can INSERT listings only for their own farmer_id.
- authenticated farmers can UPDATE only their own listings.
- Browse Produce must still work publicly.

B. agent_call_requests

Remove anonymous status update:
- Drop public_update_call_request_status policy if it exists.
- Revoke UPDATE on agent_call_requests from anon.

Restrict agent call request read/update to agent/admin users:
- Drop overly broad authenticated read/update policies if they allow all authenticated users.
- Add SELECT policy where user_profiles.role IN ('agent', 'admin').
- Add UPDATE(status) policy where user_profiles.role IN ('agent', 'admin').
- Keep status limited to:
  - pending
  - called
  - resolved

Decide insert behavior carefully:
- If the Agent Dashboard is the only place creating requests, restrict INSERT to agent/admin.
- If there is a public farmer callback form somewhere, keep public INSERT only, but do not allow public SELECT or UPDATE.
- Check the current app before deciding.
- Use the minimum safe access.

Do not allow buyer/farmer roles to read all agent call requests.
Do not allow anon to read all agent call requests.
Do not allow anon to update call request status.
