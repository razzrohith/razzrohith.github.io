PART 1 — RLS Validation Tooling / Read-Only Security Check

Goal:
Create a read-only validation tool/report that checks whether Supabase policies and grants are in the expected safe state.

Important:
- This should be read-only.
- Do not modify policies.
- Do not modify grants.
- Do not run destructive SQL.
- Do not use service_role key.
- Use only existing SUPABASE_DB_URL server-side if needed.
- Never print secrets or DB URL.

Create if useful:
- supabase/rls-validation.sql
or
- scripts/src/rls-validation.ts
or both, whichever fits the current project pattern.

The validation should check these core tables:

1. user_profiles
Expected:
- anon cannot UPDATE
- authenticated can update only safe profile fields:
  - full_name
  - phone
  - village
  - district
- authenticated cannot update:
  - role
  - id
  - created_at
- users can read/update only own profile via RLS
- frontend signup cannot create admin role

2. farmers
Expected:
- public can read verified farmers if intended
- authenticated farmer can read/insert/update own farmers row using user_id = auth.uid()
- users cannot update another farmer row
- user_id exists and is unique/linked safely

3. produce_listings
Expected:
- anon can SELECT active listings only
- anon cannot INSERT/UPDATE/DELETE
- authenticated farmer can INSERT listing only for own farmer_id
- authenticated farmer can UPDATE only safe columns
- authenticated farmer cannot update:
  - id
  - farmer_id
  - created_at
- no broad unsafe write grants

4. reservations
Expected:
- anon can INSERT guest reservation
- anon cannot SELECT reservations
- anon cannot UPDATE/DELETE reservations
- buyer can SELECT only own reservations where buyer_user_id = auth.uid()
- buyer can cancel only own pending reservation by setting status = cancelled
- buyer cannot update protected fields:
  - buyer_name
  - buyer_phone
  - listing_id
  - quantity_kg
  - payment_method
  - created_at
- farmer can SELECT/UPDATE status only for reservations on own listings
- admin can SELECT all and UPDATE status only
- no DELETE access for anon/authenticated

5. agent_call_requests
Expected:
- only agent/admin can SELECT call requests
- only agent/admin can INSERT call requests
- only agent/admin can UPDATE status
- anon cannot read/update/insert unless there is an explicit public callback form, which currently should not exist
- status limited to pending/called/resolved

6. waitlist_leads
Expected:
- anon can INSERT
- public cannot read all waitlist leads unless intentionally allowed
- no broad unsafe update/delete

Validation output:
Return a table with:
- Table
- Check
- Expected
- Actual
- Result: Pass / Fail / Warning
- Notes
