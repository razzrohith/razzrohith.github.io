PART 1 — Phase 11: Final Security/RLS Audit

Goal:
Verify the app is safe enough for MVP pilot before release-candidate testing.

Audit these tables and policies:

1. user_profiles
Expected:
- anon cannot SELECT all profiles.
- anon cannot UPDATE profiles.
- authenticated user can read own profile.
- authenticated user can update only safe fields:
  - full_name
  - phone
  - village
  - district
- authenticated user cannot update:
  - role
  - id
  - created_at
- frontend signup allows only:
  - buyer
  - farmer
  - agent
- admin role cannot be self-selected from frontend.

2. farmers
Expected:
- public can read verified farmer profile data if intended.
- farmer auth link uses user_id safely.
- authenticated farmer can insert/update only own farmer row.
- users cannot update another farmer row.
- farmer phone is not shown directly on landing page.

3. produce_listings
Expected:
- public can read active listings only.
- public cannot insert/update/delete listings.
- authenticated farmer can insert only for own farmer_id.
- authenticated farmer can update only safe columns.
- authenticated farmer cannot update:
  - id
  - farmer_id
  - created_at
- Browse Produce shows only active listings.
- Sold/out_of_stock listings are not reservable publicly.

4. reservations
Expected:
- anon can insert guest reservation only.
- anon cannot select/update/delete reservations.
- logged-in buyer can read only own reservations by buyer_user_id.
- buyer can cancel only own pending reservation.
- buyer cannot update protected fields:
  - buyer_name
  - buyer_phone
  - listing_id
  - quantity_kg
  - payment_method
  - created_at
- farmer can read/update status only for reservations on own listings.
- admin can read all reservations and update status only.
- no public buyer phone exposure.

5. agent_call_requests
Expected:
- only agent/admin can read requests.
- only agent/admin can insert requests.
- only agent/admin can update status.
- buyer/farmer/anon cannot read all agent requests.
- no paid SMS/WhatsApp API used.

6. waitlist_leads
Expected:
- anon can insert waitlist leads.
- public cannot read all waitlist leads unless intentionally allowed.
- no broad update/delete access.

Security scan:
- Search frontend for service_role.
- Search frontend for SUPABASE_DB_URL.
- Search frontend for hardcoded Supabase secret keys.
- Search for console logging of secrets.
- Search for raw paid API references:

RLS validation:

- If direct psql is blocked by DNS, run equivalent REST/client tests like last phase.
- Clearly say which method was used.
- Do not print secrets.
